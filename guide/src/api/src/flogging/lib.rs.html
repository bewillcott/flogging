26//!
27//! # FLogging
28//!
29//! The primary purpose of logging, is to facilitate fault diagnosis through the
30//! provision of specific information as, when, and from where, it is needed. This could
31//! be during development, testing, or even during production runs.
32//!
33//! ## Setting up
34//!
35//! You need to add this crate to your project:
36//! ```text
37//! $ cargo add flogging
38//! ```
39//! or add this text to the projects `Cargo.toml` file:
40//! ```text
41//! [dependencies]
42//! flogging = "0.6.0"
43//! ```
44//! ## ** Warning **
45//!
46//! Before proceeding, please read the [README.md] file.
47//!
48//! ## Features
49//!
50//! - [Levels](enum.Level.html) - There are nine (9) levels of message logging, with two (2) special ones.
51//! - [Choice](index.html#choice) - You can use either macros or methods.
52//! - [Built-in options](index.html#built-in-options) - A range of handlers and formatters.
53//! - [Customization](index.html#customization) - You can create your own handlers and/or formatters.
54//!
55//! ### Choice
56//!
57//! - [Macros](index.html#macros)
58//! - [Methods](index.html#methods)
59//!
60//! #### Macros
61//!
62//! - [Special Note](index.html#special-note)
63//!
64//! This crate has very easy to use macros. By using them, you remove a lot of the complexity
65//! from the process. Thus making it both simpler and less code cluttering, to use.
66//!
67//! - There is one macro ([`const_logger!()`]) that is used to setup a single module/file for logging.
68//! - There is one macro ([`#[logger]`][macro@logger]) that is applied as an attribute to each function/method
69//!   that you need to create log entries within.
70//! - There are nine macros that are used to actually log the messages:
71//!     - [`config!()`]
72//!     - [`entering!()`]
73//!     - [`exiting!()`]
74//!     - [`fine!()`]
75//!     - [`finer!()`]
76//!     - [`finest!()`]
77//!     - [`info!()`]
78//!     - [`severe!()`]
79//!     - [`warning!()`]
80//! - There are two helper macros:
81//!     - [`get_handler!()`]
82//!     - [`set_level!()`]
83//!
84//! Check out the [Examples](index.html#examples) below for how easy it is to get started.
85//!
86//! ##### Special Note
87//!
88//! For the macros that accept the parameter: `msg`, the following is true:
89//!
90//! - They accept parameters the same as for [`std::format!`](https://doc.rust-lang.org/std/macro.format.html)
91//!     - plain text `&str`: `("It's your time.")`
92//!     - format `&str` with interpolated variables: `("Var: {var}")`
93//!     - format `&str` with supporting parameters: `("Var: {}", var)`
94//!     - Combination of the last two: `("Vars {var1} - {}:{}", var2, var3)`
95//! - Additional Feature
96//!     - Just one or more variables without a supplied format string: `(var1, var2, var3)`
97//!     - In this case, a default format string will be used: `"{}, {}, {}"`
98//!     - The number of `"{}"` will depend on the number of parameters.
99//!     - Ideal for logging concrete instances that have very good `Display` implementations,
100//!       or you just need their data without further explanation.
101//! - Special Cases
102//!     - [entering!] and [exiting!]
103//!     - These two macros have the same features as the others,
104//!       but they may also be used _without_ any parameters. In such
105//!       a case, their defaults will be used.
106//!
107//! #### Methods
108//!
109//! - [Logger](index.html#logger)
110//! - [LoggerBuilder](index.html#loggerbuilder)
111//!
112//! Now for the coding geeks! Yes I didn't forget you lot.
113//!
114//! Though the macros are the easiest and simplest way to use this crate, those macros are just candy
115//! coating over the real workers, the methods. There are two main mods/structs in this crate, [`Logger`](struct.Logger.html)
116//! and [`LoggerBuilder`](struct.LoggerBuilder.html).
117//!
118//! ##### Logger
119//!
120//! `Logger` is the work-horse of the crate. It has all the methods for initializing each function/method
121//! for logging, and all of the message logging methods.
122//!
123//! Using the "methods" option is more complex, as-in, you have to write a lot more code, and manage it.
124//! To see how much more is involved, check-out the [`Logger`](struct.Logger.html)'s methods. There are
125//! plenty of examples throughout.
126//!
127//! - [`builder()`][Logger::builder]
128//! - [`config()`][Logger::config]
129//! - [`console_logger()`][Logger::console_logger]
130//! - [`econsole_logger()`][Logger::econsole_logger]
131//! - [`custom_logger()`][Logger::custom_logger]
132//! - [`entering()`][Logger::entering]
133//! - [`entering_with()`][Logger::entering_with]
134//! - [`exiting()`][Logger::exiting]
135//! - [`exiting_with()`][Logger::exiting_with]
136//! - [`file_logger()`][Logger::file_logger]
137//! - [`fine()`][Logger::fine]
138//! - [`finer()`][Logger::finer]
139//! - [`finest()`][Logger::finest]
140//! - [`fn_name()`][Logger::fn_name]
141//! - [`get_handler()`][Logger::get_handler]
142//! - [`has_handler()`][Logger::has_handler]
143//! - [`info()`][Logger::info]
144//! - [`level()`][Logger::level]
145//! - [`reset_level()`][Logger::reset_level]
146//! - [`set_fn_name()`][Logger::set_fn_name]
147//! - [`set_level()`][Logger::set_level]
148//! - [`severe()`][Logger::severe]
149//! - [`string_logger()`][Logger::string_logger]
150//! - [`warning()`][Logger::warning]
151//!
152//! ##### LoggerBuilder
153//!
154//! `LoggerBuilder` is used by `Logger` to provide various configuration options for setting up your logger.
155//! The available options/methods are:
156//!
157//! - [`add_console_handler()`][LoggerBuilder::add_console_handler()]
158//! - [`add_console_handler_with()`][LoggerBuilder::add_console_handler_with()]
159//! - [`add_econsole_handler()`][LoggerBuilder::add_econsole_handler()]
160//! - [`add_econsole_handler_with()`][LoggerBuilder::add_econsole_handler_with()]
161//! - [`add_custom_handler()`][LoggerBuilder::add_custom_handler()]
162//! - [`add_custom_handler_with()`][LoggerBuilder::add_custom_handler_with()]
163//! - [`add_file_handler()`][LoggerBuilder::add_file_handler()]
164//! - [`add_file_handler_with()`][LoggerBuilder::add_file_handler_with()]
165//! - [`add_string_handler()`][LoggerBuilder::add_string_handler()]
166//! - [`add_string_handler_with()`][LoggerBuilder::add_string_handler_with()]
167//! - [`set_level()`][LoggerBuilder::set_level()]
168//!
169//! And to finish:
170//! - [`build()`][LoggerBuilder::build()]
171//!
172//! These options/methods allow you a lot of flexibility in how you configure your logger. As you will typically
173//! have a different logger for each mod/file, you have a lot of control over what is logged, how it is formatted,
174//! and where it is stored/viewed. With the [`set_level()`][LoggerBuilder::set_level()] method, you can control
175//! this on a mod/file basis. Logging each file differently, or even turning logging off when you no-longer require it.
176//!
177//! **Note**
178//!
179//! As of version (0.4.0), you can only set the logging level for the logger. All handlers process every log entry
180//! that the logger accepts, based on the logger's current log level setting. This may change in a future version,
181//! allowing each handler to have its own logging level.
182//!
183//! ### Built-in options
184//!
185//! I have included a number of handlers to get you started:
186//!
187//! - [`ConsoleHandler`]
188//! - [`FileHandler`]
189//! - [`MockHandler`]
190//! - [`StringHandler`]
191//!
192//! There are also a number of formatters as well:
193//!
194//! - [`Iso8601Formatter`]
195//! - [`MockFormatter`]
196//! - [`SimpleFormatter`]
197//! - [`UnixTimestampFormatter`]
198//!
199//! ### Customization
200//!
201//! - [Custom Handler](index.html#custom-handler)
202//! - [Custom Formatter](index.html#custom-formatter)
203//!
204//! Now for the fun part - "Doing it _your_ way!!!"
205//!
206//! Though I have provided some "standard" handlers and formatters, not everyone, or every project,
207//! will want to use them. I expect there will be a need for:
208//!
209//! - sending log entries to remote system log servers,
210//! - sending log entries to another program (local or remote) for live analysis, or some other processing,
211//! - storing log entries in a specific file format (xml, json, csv),
212//! - storing log entries in a database.
213//!
214//! And I'm sure you'll come-up with more requirements at some time in the future. So, you have the option
215//! to create your own custom handlers and custom formatters. Mixing them up with the built-in ones as
216//! you need to.
217//!
218//! OK now, how do you do it. Well this is going to require some work on your part.
219//!
220//! #### Custom Handler
221//!
222//! To create a custom handler, I would suggest looking at the source code for the built-in ones, and copying
223//! the code from the one that is closest to your requirements. _Make sure that you rename as appropriate!_
224//! Then make the necessary changes, adding in your own code, to get it doing what you need.
225//!
226//! When you are ready to try-out your new custom handler, check-out these methods:
227//!
228//! - [`Logger::custom_logger()`]
229//! - [`LoggerBuilder::add_custom_handler()`]
230//! - [`LoggerBuilder::add_custom_handler_with()`]
231//!
232//! #### Custom Formatter
233//!
234//! Now for the custom formatter. This may require a bit more investigation on your part, as to the actual
235//! formatting options that are available.
236//!
237//! Firstly, this crate uses [crono] for the date/time functionality. Check
238//! out the available [specifiers]. You will need to use the formatting options from this crate for the `dt_fmt` string, of your custom
239//! formatter.
240//!
241//! Secondly, the `fmt_string` uses the format options available in accordance with [std::fmt]. Though I am
242//! actually using the [strfmt] crate to do the formatting, because it does
243//! _not_ require a 'static' string like `format!()`.
244//!
245//! Again, check-out the built-in formatters, and copy the code from the one that is closest to your
246//! requirements. _As before, renaming as necessary!_ Also, check-out the trait: [`FormatTrait`](trait.FormatTrait.html).
247//! You will need to implement it for your custom formatter, as you will notice when you look at the built-in formatters.
248//! Also, you will find that the 'provided method', [`ft_fmt()`][FormatTrait::ft_fmt], provides certain variables
249//! that you can include, via interpolation, in your `fmt_string`.
250//!
251//! Once you have got your custom formatter set up, you can then use it with:
252//!
253//! - [`LoggerBuilder::add_console_handler_with()`]
254//! - [`LoggerBuilder::add_econsole_handler_with()`]
255//! - [`LoggerBuilder::add_custom_handler_with()`]
256//! - [`LoggerBuilder::add_file_handler_with()`]
257//! - [`LoggerBuilder::add_string_handler_with()`]
258//!
259//! ## Examples
260//!
261//! This example demonstrates the use of the macros. The reason I am demoing the macros, is that I expect most
262//! people will want to use them, instead of the methods, for ease of use.
263//!
264//! Let's see what is required:
265//!
266//! 1. At the module/file level:
267//!     - `use flogging::*;`
268//!     - `const_logger!({...});`[=>][const_logger]
269//! 2. On each function/method you want to add logging to:
270//!     - `#[logger]`[=>][logger]
271//! 3. Inside each such attributed function/method:
272//!     - Any of the logging [macros]
273//!
274//! ```
275//! extern crate flogging;
276//! use flogging::*;
277//! use std::{error::Error, result::Result};
278//!
279//! // Setting up the module level logger.
280//! const_logger!({
281//!     Logger::builder(module_path!())
282//!         .add_console_handler()
283//!         .add_file_handler("test.log")
284//!         .set_level(Level::FINEST)
285//!         .build()
286//! });
287//!
288//! #[logger]
289//! fn do_something() {
290//!     entering!();
291//!
292//!     // do some work worth noting
293//!     let result = "Just something to log.";
294//!     info!("Did some work here.\n  {result}");
295//!
296//!     // ...
297//!
298//!     fine!("Bit more detail.");
299//!
300//!     if let Err(e) = error_prone() {
301//!         warning!("Error: {}", e);
302//!     }
303//!
304//!     exiting!();
305//! }
306//!
307//! #[logger]
308//! fn error_prone() -> Result<(), Box<dyn Error>> {
309//!     entering!();
310//!     let rtn = Err(Box::from("Bad day!"));
311//!     exiting!();
312//!     rtn
313//! }
314//!
315//! #[logger]
316//! fn main() {
317//!     entering!();
318//!     info!("All logging macros accept the same parameters as `std::format!(...)`");
319//!     warning!("Those same macros (info, etc.) MUST have atleast one parameter.");
320//!     config!("This is running on Fedora Linux 42.");
321//!     do_something();
322//!     info!("Job's done.");
323//!     exiting!();
324//! }
325//!
326//! ```
327//! Output:
328//! ```code
329//! flogging->main [FINER  ] Entry
330//! flogging->main [INFO   ] All logging macros accept the same parameters as `std::format!(...)`
331//! flogging->main [WARNING] Those same macros (info, etc.) MUST have atleast one parameter.
332//! flogging->main [CONFIG ] This is running on Fedora Linux 42.
333//! flogging->do_something [FINER  ] Entry
334//! flogging->do_something [INFO   ] Did some work here.
335//!   Just something to log.
336//! flogging->do_something [FINE   ] Bit more detail.
337//! flogging->error_prone [FINER  ] Entry
338//! flogging->error_prone [FINER  ] Return
339//! flogging->do_something [WARNING] Error: Bad day!
340//! flogging->do_something [FINER  ] Return
341//! flogging->main [INFO   ] Job's done.
342//! flogging->main [FINER  ] Return
343//! ```
